# ICS 4104 — Assignment 1: Customizable Load Balancer

## Overview
This project implements a customizable, Docker-based load balancer that
distributes client requests across N replicated web server containers using
consistent hashing. It demonstrates dynamic scaling (add/remove server
replicas at runtime) and automatic failure recovery (dead containers are
detected and replaced without manual intervention). Built for ICS 4104
Distributed Systems, based on the assignment brief in this repository.

    make logs    # tail the load balancer's logs
    make down    # stop the stack
    make clean   # stop, remove images/volumes, remove the docker network

The load balancer is exposed on the host at `http://localhost:5000`.

## API Endpoints
| Endpoint | Method | Purpose |
|---|---|---|
| `/rep` | GET | List current replica count and hostnames |
| `/add` | POST | Scale up: `{"n": <int>, "hostnames": [...]}` |
| `/rm` | DELETE | Scale down: `{"n": <int>, "hostnames": [...]}` |
| `/<path>` | GET | Routed to a server replica via consistent hashing |

Server-side (internal, not exposed to host):
| Endpoint | Method | Purpose |
|---|---|---|
| `/home` | GET | Returns `Hello from Server: <ID>` |
| `/heartbeat` | GET | Empty 200 OK, used by the load balancer's health checks |

## Design Choices
- **Language:** Python 3.11 + Flask for both the server and load balancer.
- **Consistent hashing:** array-based ring of M=512 slots, K=9 virtual nodes
  per server, hash functions H(i) = i^2 + 2i + 17 and
  Phi(i,j) = i^2 + j^2 + 2j + 25 exactly as specified, collisions resolved
  via linear probing.
- **Server identity:** each spawned container gets a random hostname used as
  both its Docker container name and its consistent-hash identity. A
  strictly increasing internal integer ID feeds the Phi hash function.
- **Failure detection:** a background thread polls `/heartbeat` on every
  registered server every 4 seconds. Newly spawned containers get an
  8-second grace period before being checked, to avoid false positives
  while Flask is still starting up inside the container.
- **Spawning:** the load balancer mounts the host's Docker socket
  (`/var/run/docker.sock`) and uses the `docker` Python SDK to run/stop
  sibling containers on the `net1` network, hence `privileged: true` in
  docker-compose.
- **Logging:** `PYTHONUNBUFFERED=1` is set in the load balancer's Dockerfile
  so that Python's `print()` output (used for heartbeat/failure logging)
  flushes immediately to `docker logs`, instead of sitting in an internal
  buffer.
- **Concurrency:** both Flask apps run with `threaded=True`, since the
  default single-threaded development server would otherwise serialize all
  incoming requests and misrepresent the load balancer's actual behavior
  under concurrent load testing.

## Assumptions
- Request IDs are random 6-digit integers generated by the load balancer
  itself at routing time, per Appendix B's stated assumption — the servers'

- `/add` and `/rm` without an explicit hostname list use randomly generated
  hostnames / randomly selected removals, per spec.

## Performance Analysis (Task 4)

### A-1: Load distribution at N=3, 10000 requests
![A-1 chart](analysis/results/a1_bar_chart.png)

Raw counts: SDZLOK: 8394, SOYH4Y: 1074, STRNYH: 488 (9956/10000 requests
succeeded; 44 requests failed during container startup immediately after
deployment, before all replicas had finished booting).

Observation: One server (the first one created) consistently receives
roughly 80-85% of all traffic, reproducible across repeated runs. This is
a deterministic consequence of the specified hash functions, not sampling
noise. H(i) = i^2 + 2i + 17 simplifies to (i+1)^2 + 16, and since squaring
modulo a power of two (M=512=2^9) only ever produces a small fixed set of
results, H(i) can only ever land on approximately 87 of the 512 ring slots
(~17%), regardless of how many distinct request IDs are hashed. Separately,
Phi(i,j) = i^2 + j^2 + 2j + 25 places the first server's virtual nodes at
the lowest slot numbers on the ring (since the i^2 term is smallest for
i=1), and these happen to sit clockwise-nearest to a disproportionate share
of that fixed 87-slot landing set. The result is a systematic, repeatable
skew toward whichever server was created first, independent of request
volume. This directly motivates the A-4 experiment below.

### A-2: Scalability, N=2 to 6
![A-2 chart](analysis/results/a2_line_chart.png)

Raw data:
    N=2 -> avg 5000.0/server, counts={SDZLOK: 8713, SOYH4Y: 1287}
    N=3 -> avg 3333.3/server, counts={SDZLOK: 8522, SOYH4Y: 977, S54KIW: 501}
    N=4 -> avg 2500.0/server, counts={SDZLOK: 8308, S54KIW: 315, SOYH4Y: 917, SHW4KL: 460}
    N=5 -> avg 2000.0/server, counts={SDZLOK: 8130, S54KIW: 314, SHW4KL: 484, SOYH4Y: 992, S2AOA9: 80}
    N=6 -> avg 2000.0/server, counts={SDZLOK: 7820, SOYH4Y: 952, S6ISJ5: 379, S54KIW: 338, SHW4KL: 511}

Observation: average load per server decreases as N grows, confirming the
system does scale horizontally as designed — the load balancer correctly
spreads work across more replicas as N increases. However, the same
first-server skew from A-1 persists at every N: SDZLOK, the server created
first in this run, retains 78-87% of total traffic regardless of how many
additional servers are added. Notably, at N=6 only 5 distinct servers
appear in the results — one replica received zero requests out of 10000,
a complete load-balancing failure for that server. This shows that
consistent hashing with these specific hash functions does not achieve
even load balancing on its own; the theoretical scalability of the
architecture (more replicas = more total capacity) is present, but
per-replica fairness is compromised by the hash functions' structural
bias. (Note: the "avg load" figures above are computed over the number of
servers that received at least one request, not the configured N — at
N=6 this silently divides by 5 rather than 6, which is why the N=5 and
N=6 averages both read 2000.0.)

### A-3: Failure recovery

every 4 seconds, with an 8-second grace period for newly spawned
containers) detected the failure and logged:

### A-4: Modified hash functions
Isolated implementation in `experiments/alt_hash_loadbalancer/` (kept
separate from the graded `loadbalancer/` to avoid altering the
spec-required hash functions there). Replaced the arithmetic H(i) and
Phi(i,j) with MD5-based hashing:
    H(i) = int(md5(str(i)), 16) % M
    Phi(i,j) = int(md5(f"{i}-{j}"), 16) % M
